QObject
    Q_OBJECT enables Qt’s meta-object system. It allows features like signals and slots, properties, runtime type information, 
    and dynamic method invocation. It works by triggering Qt’s Meta-Object Compiler, which generates additional C++ code. Without it, 
    these runtime features won’t function
    Almost everything in Qt inherits from QObject.
    -object hierarchy
    -memory management
    -signals and slots
    -event handling
    -properties

    - Q_OBJECT is a macro that enables Qt’s Meta-Object System for a class.
    - it is required for any class to have features like signal and slot, Q_PROPERTY,qobject_cast,QObject::metaObject(),
        dynamic method invocation(QMetaObject::invokeMethod(obj,"start"));,translations
    - C++ does not support runtime information about methods, signals and properties.
            - QT solves by using Q_OBJECT macro
            - running moc(meta object compiler) during build.
            - generating extra C++ code that provides runtime metadata.

    class qtClass: public QObject {
        Q_OBJECT
    }

    -it expands to (moc generates this code)
        - declarations of meta-object data
        - virtual functions like:
            metaObject()
            qt_metacall()
            qt_metacast()
        - static meta object tables.

    Role of MOC (meta object compiler)
        -it scans the header file
        - Looks for Q_OBJECT
        - generates a .moc C++ file.
        - compiler links it 
    
    qMetaObject
    -QMetaObject is Qt’s runtime representation of a QObject-derived class. It stores metadata such as class name, 
        inheritance, signals, slots, properties, and enums. It is generated by MOC and 
        enables Qt’s reflection-like features such as signal-slot dispatch, runtime method invocation, 
        and qobject_cast.
    - Every class that has Q_OBJECT has one static QMetaObject instance.
    - One per class.
    - generated at compile time by MOC.
    - It stores metadata about the class ( class name, inheritance hierarchy, signals, slots, properties, methods, enumerations)
    - Used for:
        RTTI
        qobject_cast
        Signal inheritance

QML ↔ C++ integration internals
    QML and C++ communicate through Qt’s meta-object system. 
    C++ objects expose properties, methods, and signals using Q_OBJECT, Q_PROPERTY, and Q_INVOKABLE. 
    The QML engine uses QMetaObject to inspect metadata, invoke methods, bind properties, 
    and react to signals. All communication is reflection-based, type-safe, and optimized 
    with cached lookups and dependency tracking


    QML (UI, JS)
    ↓
    Qt QML Engine (QQmlEngine)
    ↓
    Qt Meta-Object System
    ↓
    C++ QObject classes


    Core components involved
        | Component      | Role                       |
        | -------------- | -------------------------- |
        | QQmlEngine     | Loads & runs QML           |
        | QQmlContext    | Exposes C++ objects to QML |
        | QObject        | Base of all C++ objects    |
        | QMetaObject    | Runtime metadata           |
        | QMetaProperty  | Property access            |
        | QMetaMethod    | Method calls               |
        | QMetaType      | Type conversion            |
        | Binding Engine | Tracks dependencies        |

Signals and slots
    - “Qt signals and slots work via the meta-object system. 
        Signals are declared in the class with Q_OBJECT and MOC generates code 
        that calls QMetaObject::activate. When a signal is emitted, it looks up all connected slots from the sender’s connection list. 
        Slots are invoked either directly or via the event system depending on connection type. 
        Parameters are marshalled using QMetaType. Threading is handled automatically with Direct, Queued, Auto, and BlockingQueued connections.”
    -used for communication between 2 objects.

    class Button: public QObject{
        Q_OBJECT
        public:
            signals;
                void clicked();
            public slots:
                void onClicked();
    }

    - signal : emitted by an object to announce an event.
    - slot : function that receives signals
    - connect : binds signals and slots.

    connect(button,&Button::clicked,receiver,&Receiver::onClicked);

    - Signals have no implementation.(just decalaration).
    - the compiler does not generate call code.
    - moc generates code that implements:
        - signal dispatch
        - slot registration
        - parameter registration.

    step1 MOC Processing
        class Button : public QObject {
            Q_OBJECT
        signals:
            void clicked(int x);
        };

        MOC generates 
        1.static tables describing:
            class name, signals and slots, parameter types, method indices.
        2. signal code that calls QObject::metaObject()->activate()

    step2 Connect internals
        connect(sender,&Button::clicked,receiver,&Receiver::onClicked);

        1. iternally qt checkts slot/signal compatibiltly.(same number and compatible types).
        2. adds an entry to QObjects connection list.
        struct Connection {
            QObject *sender;
            int signalIndex;
            QObject *receiver;
            int slotIndex;
            Qt::ConnectionType type; // Direct, Queued, BlockingQueued
        };
        3. stored in senders connection table.

    
    step 3: signal emission
        - when clicked() is called.
            the moc-generated calls 
            QMetaObject::activate(this,metaObject,signalIndex,argv)
        - active does
            - iterate through all connections for this signal.
            - for each connection type  
                - determine connection type.
                - if DirectConnection : call slot immediately.
                - if QueuedConnection : post QEvent to receivers event queue.
                - if BlockingQueuedConnection: wait for receiver thread.
        - parameters are marshalled via void pointers(argv array)
            -automatic  conversion of basic types(int, QString,QObject*)

    step 4: slot invocation
        -direct connection.
            - slot is called like a normal function.
            - happpens in senders thread.
            
        - queued connection
            - sender posts a QEvent to receivers thread.
            - receivers event loop picks it up.
            - calls slot in receivers thread.
        
        - auto connection
            -default behavior:
                -if same thread -> direct
                -if different thred -> queued

    step 5 : parameter marshalling
        signals:
            void valueChanged(int x, QString s);
        slots:
            void handleValue(int x, QString s);

        internally
            - moc stores type names in meta-object
            - QMetaType is used to copy/convert parameters.
            - Queued connection copies arguments safely.
            
    step 6 thread handling

    step 7 : disconnecting signals.
        - removes connection from senders table.
        - can disconnect all signals, specific signals, or specific receiver.


    How signals are stored internally
        -each QObject has a connection list.
        - each conneciton contains:
            signal index
            receiver pointer.
            slot index
            connection type.
        - stored in a linked list or vector
        - QMetaObject provides signal/slot mapping.


Q_DECLARE_METATYPE(MyType)
    -Lets Qt know at compile-time that MyType is a type that can be used in QVariant and in the meta-object system.
    -Must be placed in the header of your type, usually after the class declaration

    class MyType {
    public:
        int a;
        QString name;
    };

    Q_DECLARE_METATYPE(MyType)


qRegisterMetaType<MyType>("MyType")
    -Purpose:
        Registers the type at runtime in Qt’s meta-type system.
    Needed for:
        Passing the type through signals/slots across threads (queued connections)
        Dynamic property systems (setProperty, property)
        QVariant conversions



    Q_DECLARE_METATYPE declares a type to the Qt meta-object system at compile-time so it 
    can be stored in QVariant and used in signals, slots, and dynamic properties. 
    qRegisterMetaType registers the type at runtime, giving it a type ID and allowing it to be 
    safely passed in queued signals or stored dynamically

QVariant is a generic container in Qt that can store values of various types at runtime.

| Feature         | QString                   | QByteArray                     |
| --------------- | ------------------------- | ------------------------------ |
| Stores          | Unicode text              | Raw bytes                      |
| Encoding        | UTF-16 internally         | 8-bit (UTF-8, ASCII, binary)   |
| Usage           | UI, text processing       | File/network data, conversions |
| API             | Rich string methods       | Byte array manipulations       |
| Copy efficiency | Copy-on-write             | Copy-on-write                  |
| Conversion      | Can convert to QByteArray | Can convert to QString         |


QString is Qt’s Unicode string class stored as UTF-16 with copy-on-write for efficient text handling. 
QByteArray is a raw byte container for binary or encoded data, also using copy-on-write. 
QString is used for text/UI, while QByteArray is used for network, file I/O, or low-level binary data

Two way binding
    class Backend: public QObject {
        Q_OBJECT
        Q_PROPERTY(QString value READ value WRITE setValue NOTIFY valueChanged)

        signals:
            void valueChanged();
        
        private:
            QString m_value;

        public: 
            QString value() const (m_value;)

            void setValue(const QString& v) {
                if(m_value == v) return;
                m_value = v;
                emit valueChanged();
            }
    }
